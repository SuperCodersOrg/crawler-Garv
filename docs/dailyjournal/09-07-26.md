# Daily Design Journal · Day 9

## Section 1 – Specific Bug

The biggest issue today was while implementing the `URLNormalizer`. One of the normalization test cases kept failing even though most of the implementation was working.

```cpp id="1e9hwp"
98% tests passed, 1 tests failed out of 65

The following tests FAILED:
33 - URLNormalizerTest.KeepNonDefaultPort (Failed)
```

After debugging it, I found that I never reset `result.hasport` to `false` when the URL didn't contain a port, so the previous value was being carried over and causing the test to fail.

Later, while implementing the HTML parser, many unit tests kept failing because of small syntax mistakes, logical bugs and even one incorrect test case that I had written myself.

## Section 2 – Failed Attempt

For the URL normalizer I first started implementing relative URL resolution, but quickly realised I needed stack-like operations while processing `"."` and `".."` path segments. Instead of writing another data structure, I decided to extend my own `DynamicArray` by adding `back()`, `const back()` and `pop_back()`. That made the normalizer implementation much cleaner.

While implementing the KMP algorithm, I got stuck understanding how the LPS (Longest Prefix Suffix) array was actually built. Reading the algorithm wasn't enough, so I watched a few visual explanations and stepped through the algorithm manually before writing the code. After understanding how the fallback works, implementing it became much easier.

The HTML parser also took longer than expected. Initially I thought the extraction logic was correct because it worked on simple HTML, but once I started running unit tests many edge cases failed. I spent quite a bit of time debugging before realising that one of the failing tests was actually written incorrectly. After fixing both the implementation and that test case, everything passed.

## Section 4 – Learning Reflection

Today was mostly about building the remaining core components of the crawler and making them work together.

While implementing the URL normalizer, I realised that sometimes existing data structures need to be extended instead of creating completely new ones. Adding stack operations to my `DynamicArray` was enough to support relative path resolution, and those functions will probably be useful in other places too.

Understanding KMP was another important part of today. Earlier I only knew that it was faster than naive string matching, but after implementing the LPS array myself I finally understood *why* it avoids unnecessary comparisons. Seeing the fallback process step by step made the algorithm much clearer than simply memorizing it.

The HTML parser also reminded me that testing with only small examples isn't enough. It worked on simple HTML snippets but started failing when I introduced more realistic cases. Running it on an actual website helped me verify that it could extract links correctly instead of only passing artificial test cases.

By the end of the day I had implemented the main parts of the `URLNormalizer`, extended my `DynamicArray` with stack operations, completed the KMP pattern matcher with comprehensive unit tests, and finished the HTML parser that extracts links using KMP. All of the unit tests passed, and I also verified the HTML parser using the rendered HTML from a real website to make sure it behaved correctly outside of the test suite.
