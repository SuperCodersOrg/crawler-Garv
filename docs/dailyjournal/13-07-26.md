# Daily Design Journal · Day 13

## Section 1 – Specific Bug

The main issue today was integrating all the crawler components together after updating the CDP implementation. The crawler started successfully, but the filtering pipeline wasn't behaving correctly and fragments were still being treated as different URLs.

While testing, I also noticed configuration files were not being loaded correctly.

```text
Could not open config/blockeddomains.txt

```

Later, after running the crawler for some time, another issue appeared while restarting Chrome.

```text
bind() returned an error:
Only one usage of each socket address (protocol/network address/port) is normally permitted. (0x2740)
```

In the second half of the day I also struggled to connect MySQL with my C++ project because the MySQL connector wasn't working properly with the MSYS2 UCRT toolchain I had been using.

## Section 2 – Failed Attempt

The first thing I tried was simply plugging the new CDP module into the crawler and expecting everything to work. It compiled, but once I started crawling I noticed duplicate URLs, fragments not being removed correctly and some filter rules being ignored.

I went back and reviewed both the Quick Filter and Link Filter. After fixing a few logical mistakes and removing fragments before inserting URLs into the `SeenStore`, duplicate entries reduced significantly. I also added an option to either stay within the same domain or allow external domains depending on the crawler configuration.

Later I spent quite a while trying to use the official MySQL Connector/C++ with my existing UCRT setup. I tried different versions and configurations but couldn't get it working reliably. Instead of changing the entire project toolchain, I switched to the MariaDB connector, which worked with the rest of my build environment. After that I implemented the storage layer and successfully stored crawled pages inside the database.

Another small issue was that the updated CDP wasn't returning all the information I expected, especially the crawl timestamp. I modified the crawler to capture and store the date and time along with the crawled data before inserting it into the database.

## Section 4 – Learning Reflection

Today felt more like system integration than writing individual components.

Most of the modules already worked on their own, but putting everything together exposed issues that weren't visible during unit testing. Small things like fragment handling or configuration loading became much more important once the crawler started processing hundreds of links.

I also learnt that choosing compatible libraries is sometimes more important than choosing the "official" ones. I spent quite a bit of time trying to make the MySQL connector work with UCRT, but eventually realised it wasn't worth fighting the toolchain. Switching to the MariaDB connector solved the problem much faster and still allowed me to use a MySQL database.

Another thing I understood today was that storing crawled pages isn't just about saving HTML. Metadata like crawl time and URL state is equally important because it will later help decide whether a page needs to be revisited or updated.

By the end of the day the crawler was successfully running with the updated CDP integration, the Quick Filter and Link Filter bugs were fixed, fragment handling and same-domain filtering were added, the `SeenStore` was working correctly, and I had integrated a MariaDB-based storage layer with two database tables for storing crawled pages and metadata. The crawler could now fetch rendered pages and insert the data into the database successfully.
