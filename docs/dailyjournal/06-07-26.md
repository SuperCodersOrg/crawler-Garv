# Daily Design Journal · Day 6

## Section 1 – Specific Bug

The main issue today wasn't a compiler error but an implementation problem.

Initially I planned to implement the crawler's `fetch()` function by finding JavaScript requests and calling the APIs directly. After testing it on a few websites I realised that some pages were making more than **250 network requests**, which would mean making hundreds of API calls just to reconstruct one page. That approach clearly wasn't practical.

While implementing the CDP version I also ran into another issue where my executable wasn't using the correct `libcurl` DLL.

Because XAMPP's `curl` was appearing earlier in the system `PATH`, Windows was loading that dependency instead of the MSYS2 version, causing the executable to fail.

## Section 2 – Failed Attempt

My first idea was to avoid opening a browser completely. I thought I could inspect the page, collect all the JavaScript API requests and fetch everything myself. It sounded simpler at first, but after testing a few modern websites I realised it doesn't scale at all. Many pages make hundreds of requests and some of them depend on previous responses, authentication or JavaScript execution.

After that I built a small prototype using Playwright. It actually worked really well and gave me fully rendered HTML, but since Playwright depends on Node.js I didn't want my C++ crawler to depend on another runtime just for rendering pages.

That pushed me towards Chrome DevTools Protocol (CDP). Learning CDP took some time because there were multiple components involved—starting Chrome, discovering targets, opening a WebSocket connection and sending browser commands. I implemented each part separately before combining everything into a working fetch function.

Another small issue was Chrome opening a new tab every time I requested a page. After reading the documentation and experimenting a bit, I started maintaining a session instead of creating a new target every time, and also added a headless option so Chrome could run in the background.

## Section 4 – Learning Reflection

Today completely changed how I was thinking about the fetch stage of the crawler.

Initially I assumed downloading a page meant collecting the API calls made by JavaScript and replaying them myself. After experimenting with a few websites, I realised that modern web applications make far too many requests and often depend on JavaScript execution, browser state and asynchronous events. Trying to recreate all of that manually would make the crawler much more complicated than necessary.

Using Playwright showed me that rendering the page inside a browser solves this problem, but it also made me think about project dependencies. Since I wanted the crawler to stay primarily in C++, I started looking for a browser automation approach that didn't require Node.js. That's when I found Chrome DevTools Protocol.

While implementing CDP I also understood that browser automation is built from several smaller pieces rather than one library. Launching Chrome, discovering the debugging endpoint, opening a WebSocket connection and sending protocol commands all happen independently before they work together.

By the end of the day I had paused work on the second design proposal until I had a working fetch solution, built the basic CDP components, successfully downloaded rendered HTML through Chrome, and updated my project direction based on what I learnt. The implementation is working now, although it is slower than I would like, so the next step will be looking for ways to optimize it without changing the overall design.
