## Section 4 – Learning Reflection

Today was mostly about turning the overall crawler design into something that could actually be implemented.

Finishing the design proposal made me think more carefully about how each component depends on the others instead of designing them independently. Drawing the architecture diagrams also helped me notice a few interactions that weren't obvious when they were only written as text.

While implementing the Queue and Frontier, I realised that project structure becomes more important as the project grows. The code itself wasn't difficult, but integrating components across submodules and making CMake build everything correctly took more time than writing the actual classes.

Writing more unit tests for the URL normalizer also reminded me that passing a few obvious test cases doesn't necessarily mean a function is correct. The malformed URL case only appeared because I intentionally started adding edge cases instead of testing only valid inputs. After fixing the validation logic and extending the tests, I felt much more confident that the function behaved the way I expected.

By the end of the day I had completed the third version of the design proposal with architecture diagrams, finished the browser rendering documentation for the CDP-based fetch module, implemented the `Queue`, `Frontier` and `URLDepth` components along with their unit tests, integrated them into the project using CMake, and started implementing the `URLNormalizer`, including additional test cases to improve its validation logic.
